analyze-issue.md
 워크플로우를 사용하기 위한 방법과 준비 사항은 아주 간단합니다.

1. 사용 전 준비 사항
별도로 설치하거나 복잡하게 세팅할 것은 거의 없습니다.


필수: 분석할 이슈(Issue)의 내용이 있어야 합니다. (GitHub 이슈 URL이나 텍스트 내용)
권장: 에이전트(저)에게 해당 프로젝트의 코드에 대한 읽기 권한이 있어야 합니다. (현재 저는 이미 코드를 보고 있으므로 준비 완료)
2. 사용 방법 (명령 내리기)
저에게 다음과 같이 말씀만 하시면 됩니다.

예시 1 (가장 추천):

"7번 이슈 분석해줘. (analyze-issue 워크플로우 써서)"
-- 깃허브 이슈 탭 사용 : https://github.com/wisecomm/dog-love/issues

예시 2 (구체적으로):

"이 버그 좀 봐줘. 사용자가 로그인 안 된다고 하는데 원인 찾아줘." (제가 자동으로 워크플로우를 떠올리고 사용자 추측을 배제한 채 분석을 시작합니다)

3. 실제 작동 원리
님이 명령을 내리면 제가 내부적으로 다음 단계를 수행합니다:

📖 이슈 읽기: "로그인이 안 돼요..." (사용자 불만 접수)
🕵️‍♂️ 팩트 체크: "사용자는 DB 문제라고 했지만 무시하자. 코드를 보자."
🔍 코드 추적: LoginController -> AuthService -> UserRepository 순서로 코드를 쭉 읽어봅니다.
💡 결론 도출: "찾았다! DB가 아니라 토큰 만료 시간 설정이 0초로 되어 있네."
📝 보고: "DB 문제가 아닙니다. 토큰 설정 문제입니다. 이렇게 고치면 됩니다."
지금 바로 테스트해보고 싶으신 이슈가 있다면 번호나 내용을 알려주세요!

audit-changelog.md
 사용법과 준비 사항을 정리해 드립니다.

1. 사용 전 준비 사항
이 워크플로우는 Git 기록을 기반으로 작동하므로, 다음 조건이 필요합니다.

Git 태그(Tag): 이전 버전이 어디까지인지 알아야 하므로, 적어도 하나 이상의 Git 태그(예: v1.0.0)가 있어야 합니다. (없다면 기준점이 없어 처음부터 다 뒤져야 함)
CHANGELOG.md 파일: 변경 내역을 기록할 파일이 프로젝트 내에 존재해야 합니다. (없으면 새로 만들어야 함)
Git 커밋 기록: 릴리즈 이후 쌓인 커밋들이 있어야 검사할 거리가 있겠죠?
2. 사용 방법 (명령)
다음과 같이 말씀해주시면 됩니다.

"이번 릴리즈 준비하게 체인지로그 감사(Audit) 좀 해줘."

3. 실제 작동 (내부 프로세스)
제가 이 명령을 받으면 다음 순서로 움직입니다.

🏷️ 기준점 찾기: git tag 명령어로 가장 최근 배포 버전(예: v1.2.3)을 찾습니다.
📜 목록 뽑기: v1.2.3 이후부터 지금까지 쌓인 모든 커밋 리스트를 뽑습니다.
🕵️‍♂️ 대조하기:
커밋: "로그인 버튼 수정함"
CHANGELOG: (확인 중...) "어? 로그인 관련 내용이 없네?"
🚨 적발 및 보고:
"총 5개의 커밋이 CHANGELOG에 누락되었습니다."
"3번 커밋은 bugfix인데 기록이 없습니다. 추가할까요?"
요약
준비물은 "Git으로 관리되는 프로젝트" 하나면 충분합니다. 명령만 내리시면 제가 "빠진 숙제 검사하듯이" 꼼꼼하게 찾아드립니다.

review-pr.md
 워크플로우를 사용하기 위한 방법과 준비 사항입니다.

1. 사용 전 준비 사항
PR (Pull Request): 검토할 PR이 생성되어 있어야 합니다. (GitHub에 등록된 상태)
GitHub CLI (gh): 에이전트(저)나 실행 환경에 gh 명령어가 설치되어 있어야 PR 정보를 읽어올 수 있습니다. (저는 이미 사용 가능합니다)
권한: 해당 레포지토리(Repository)에 접근하여 PR 내용을 읽을 수 있는 권한이 필요합니다.
2. 사용 방법 (명령)
가장 간단하게 저에게 이렇게 말씀하시면 됩니다.

"PR #123 리뷰해줘." (또는 링크를 주셔도 됩니다: "이 PR 리뷰 좀: https://github.com/owner/repo/pull/123")

3. 실제 작동 (내부 프로세스)
이 워크플로우에 따라 저는 다음과 같이 움직입니다.

📄 문서 독해: PR 제목과 설명을 읽고 "무엇을 왜 바꾸려는지" 파악합니다.
🔍 코드 정밀 검사: 변경된 파일(Diff)을 한 줄 한 줄 뜯어봅니다.
Good: "오, 이 함수 분리는 깔끔하네요."
Bad (BLOCKER): "잠깐, 여기서 null 체크 안 하면 앱 뒈지는데요?"
Ugly (NIT): "변수명 temp 말고 userIndex가 어떨까요?"
🧪 테스트 확인: "기능은 짰는데 테스트 코드는 왜 없죠?" (테스트 누락 지적)
📝 리포트 제출: 최종적으로 [승인 / 수정 필요] 의견과 함께 상세한 리뷰 리포트를 작성해 드립니다.
핵심 포인트
그냥 "코드 어때?"라고 묻는 것보다, 이 워크플로우를 타면 "테스트는 있는지, 기존 기능 망가뜨리는 건 없는지" 기계적인 체크리스트를 통과해야 하므로 훨씬 깐깐하고 안전한 리뷰가 가능합니다.
